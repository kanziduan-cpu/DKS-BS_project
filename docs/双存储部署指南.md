## 架构说明

### 双存储架构优势

```
┌─────────────────┐
│   STM32 设备     │
└────────┬────────┘
         │ MQTT
         ▼
┌─────────────────────────────────┐
│    Node.js 服务器 (双存储管理器)   │
│                                 │
│  ┌──────────┐        ┌────────┐ │
│  │ SQLite   │◄──────►│Supabase│ │
│  │ (本地)   │ 双写同步│ (云端) │ │
│  └──────────┘        └────────┘ │
│       ▲                    ▲    │
│       │                    │    │
│  实时查询              历史分析   │
└─────────────────────────────────┘
         │
         │ WebSocket
         ▼
┌─────────────────┐
│  Android APP     │
└─────────────────┘
```

### 存储模式

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **dual** | SQLite + Supabase 双存储 | **推荐：生产环境** |
| **local** | 仅 SQLite | 开发测试 |
| **cloud** | 仅 Supabase | 特殊需求 |

### 数据流程

1. **数据写入**：同时写入 SQLite 和 Supabase
2. **实时查询**：从 SQLite 读取（快速）
3. **历史分析**：从 Supabase 读取（完整）
4. **同步队列**：Supabase 失败时数据入队列，定时重试

- **anon public key**: `eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...`

### 3. 创建数据库表

在 Supabase 的 SQL Editor 中执行 `cloud-server/create-supabase-tables.sql`：

```sql
-- 传感器数据表
CREATE TABLE sensor_data (
    id BIGSERIAL PRIMARY KEY,
    device_id TEXT NOT NULL,
    timestamp TIMESTAMPTZ DEFAULT NOW(),
    temperature REAL,
    humidity REAL,
    co REAL,
    co2 REAL,
    formaldehyde REAL,
    water_level REAL,
    vibration INTEGER,
    tilt REAL,
    gyroscope JSONB
);

-- 设备状态表
CREATE TABLE device_status (
    device_id TEXT PRIMARY KEY,
    ventilation BOOLEAN DEFAULT false,
    dehumidifier BOOLEAN DEFAULT false,
    servo_angle INTEGER DEFAULT 0,
    alarm BOOLEAN DEFAULT false,
    last_updated TIMESTAMPTZ DEFAULT NOW()
);

-- 报警记录表
CREATE TABLE alarms (
    id BIGSERIAL PRIMARY KEY,
    device_id TEXT NOT NULL,
    alarm_type TEXT NOT NULL,
    message TEXT,
    severity TEXT,
    acknowledged BOOLEAN DEFAULT false,
    timestamp TIMESTAMPTZ DEFAULT NOW()
);

-- 创建索引
CREATE INDEX idx_sensor_device_time ON sensor_data(device_id, timestamp);
CREATE INDEX idx_sensor_time ON sensor_data(timestamp DESC);

-- 启用实时功能
ALTER PUBLICATION supabase_realtime ADD TABLE sensor_data;
ALTER PUBLICATION supabase_realtime ADD TABLE alarms;
```

        proxy_set_header Upgrade $http_upgrade;

        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;

        proxy_pass http://localhost:3001;

        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
    }

systemctl restart nginx

## 故障排查

### 1. Supabase 连接失败

**症状**：日志显示 "Supabase 连接失败"

**解决方案**：
1. 检查 `.env` 文件中的 URL 和 Key 是否正确
2. 确认 Supabase 项目是否正常运行
3. 检查网络连接

### 2. 数据同步失败

**症状**：数据未同步到 Supabase

**解决方案**：
```bash
# 查看同步状态
node sync-monitor.js stats

# 手动触发同步
curl -X POST http://localhost:3000/api/sync/now

# 查找未同步数据
node sync-monitor.js find-unsynced
```

### 3. 端口被占用

**症状**：启动失败，提示端口被占用

**解决方案**：
```bash
# 查找占用端口的进程
netstat -tlnp | grep 3000

# 杀死进程
kill -9 <PID>

# 或修改配置文件中的端口
```

### 4. SQLite 数据库损坏

**症状**：查询 SQLite 时报错

**解决方案**：
```bash
# 备份现有数据库
cp data/sensor_data.db data/sensor_data.db.backup

# 删除并重新初始化
rm data/sensor_data.db
pm2 restart warehouse-server
```

### 5. 内存不足

**症状**：服务频繁重启

**解决方案**：
```bash
# 增加 PM2 内存限制
pm2 start server-dual-storage.js --name warehouse-server --max-memory-restart 500M

# 或优化 SQLite WAL 模式
# 在 storage-manager.js 中已启用
```

## 论文写作要点

### 技术亮点

1. **双层存储架构**：SQLite 本地缓存 + Supabase 云端存储
2. **双写同步机制**：数据同时写入两个存储，保证一致性
3. **智能查询路由**：实时查询走 SQLite，历史分析走 Supabase
4. **断线续传机制**：同步队列保证数据不丢失
5. **自动化维护**：定时同步和清理，减少人工干预

### 论文表述示例

> 本系统采用"本地缓存 + 云端存储"的双层架构，基于 Node.js 构建双存储管理器，通过 SQLite 提供毫秒级实时查询响应，利用 Supabase 实现云端数据共享和跨设备访问。系统采用双写同步机制，确保数据一致性；通过智能查询路由，根据请求类型自动选择最优存储，兼顾性能与可靠性；同步队列机制在网络异常时缓存数据，保证数据完整性。
